VISION & SCOPE DOCUMENTUnievents - University Event Planning, Management & Digital Archive Platform
Project VisionUnievents is designed as a centralized, structured, and secure digital platform for managing the complete lifecycle of university events.The system aims to replace fragmented event handling practices with a unified institutional solution that ensures consistency, accessibility, control, and long-term preservation of event data.The long-term vision is to transform university events from temporary activities into structured institutional records maintained within a controlled digital archive.
Core ProblemUniversity event information is currently dispersed across multiple platforms such as temporary event websites, social media pages, shared cloud drives, and manual documentation systems.These methods create:
Data fragmentation
Duplication of information
Poor traceability
Loss of historical records
Limited administrative oversight
Lack of standardized documentationThere is no centralized system that manages event creation, lifecycle progression, and long-term archival in a structured manner.
Primary ObjectiveTo develop a centralized web-based platform that enables:
Structured event planning
Controlled event management
Role-based access enforcement
Lifecycle-based event handling
Secure data storage
Searchable digital archival of all university eventsThe system must serve both operational requirements (event execution) and institutional requirements (historical preservation and oversight).
Functional ScopeThe system will include:
Authentication & Access Control
Secure user registration and login
JWT-based authentication
Role-based authorization (Student, Organizer, Admin)
Backend-enforced permission control
Event Management
Event creation, editing, deletion
Ownership tracking
Role-restricted modification
Controlled publishing process
Event Lifecycle Management
Draft state
Published state
Archived state
Enforced transition rules
Read-only archival state
Data Management
Structured event schema
Metadata storage
Timestamps and audit tracking
Centralized database storage
Archive System
Long-term event preservation
Searchable archive
Filterable event retrieval
Organized historical record maintenance
Administrative Control
System-level visibility
Moderation authority
Role supervision (future-ready design)
Non-Functional ScopeSecurity
Backend as source of truth
Password hashing
Token-based authentication
Route protection middleware
Scalability
Modular backend architecture
Expandable schema design
Future media storage compatibility
Maintainability
Clean folder structure
Defined API contracts
Clearly separated frontend and backend layers
Out of ScopeThe following are intentionally excluded:
Payment gateways
Ticket booking systems
Financial accounting modules
Native mobile applications
Enterprise ERP integration
Paid cloud storage solutions
System Design Principles
Backend authority over frontend
Role-based restriction enforcement
Data integrity and consistency
Lifecycle-driven event control
Institutional archival preservation
Minimal trust in client-side logic
Target UsersStudents
View-only access to published events
Access to archive
Organizers
Create and manage owned events
Control event lifecycle within permitted rules
Administrators
Full system-level oversight
Override capabilities
Supervisory control
Constraints
Academic semester timeline
Free-tier hosting and database usage
No paid third-party services
Limited development resources
Long-Term DirectionThe platform is architected not merely as a semester project but as a foundational institutional system that can evolve into a scalable university-wide event lifecycle and archival management solution.The system is designed to be extendable, modular, and adaptable for multi-institution deployment if required in the future.

Unievents - Backend Contract & Execution Model
Version 1 (Strict RESTful Design)
1. Purpose
This document defines:
All backend endpoints
Request and response structure
Authentication requirements
Role enforcement per route
Middleware execution flow
Error response standards
This document acts as the formal contract between frontend and backend.
2. Base Configuration
Base URL/api/v1
Authentication MethodJWT via Authorization header
Header FormatAuthorization: Bearer 
All protected routes require valid JWT.
3. Authentication Routes
3.1 Register User
POST /api/v1/auth/register
AccessPublic
Request Body
{  "name",  "email",  "password",  "role": "student | organizer",  "enrollmentId": "(optional)"}
Response (201 Created)
{  "success": true,  "message": "User registered successfully"}
Rules
Admin role cannot be registered publicly
Email must be unique
Password must be hashed
3.2 Login
POST /api/v1/auth/login
AccessPublic
Request Body
{  "email",  "password"}
Response (200 OK)
{  "success": true,  "token": "<jwt_token>",  "user": {    "id",    "name",    "role"  }}
4. Event Routes
Resource: eventsBase Path: /api/v1/events
4.1 Create Event
POST /api/v1/events
AccessOrganizer, Admin
Request Body
{  "title",  "description",  "category",  "eventDate",  "startTime",  "endTime",  "venue",  "tags": "(optional)"}
Response (201 Created)
{  "success": true,  "data": "eventObject"}
4.2 Get All Published Events
GET /api/v1/events
AccessPublic (or authenticated if policy requires)
Query Parameters (optional)
?year=2026
?category=technical
?status=published
Returns only:
published events
not soft-deleted
Response (200 OK)
{  "success": true,  "count": "<number>",  "data": ["events"]}
4.3 Get Single Event
GET /api/v1/events/:id
AccessDepends on status:
draft  owner or admin
published  all
archived  all
soft-deleted  admin only
Response (200 OK)
{  "success": true,  "data": "eventObject"}
4.4 Update Event
PUT /api/v1/events/:id
AccessOrganizer (owner only), Admin
Rules
Cannot modify archived event (unless admin override)
Cannot modify soft-deleted event
Lifecycle validation required
Response (200 OK)
{  "success": true,  "data": "updatedEvent"}
4.5 Change Event Status
PATCH /api/v1/events/:id/status
AccessOrganizer (owner), Admin
Request Body
{  "status": "published | archived"}
Rules
Must validate allowed transitions
If approval enabled, publishing requires admin
Response (200 OK)
{  "success": true,  "data": "updatedEvent"}
4.6 Soft Delete Event
DELETE /api/v1/events/:id
AccessOrganizer (owner), Admin
Behavior
Sets isDeleted = true
Records deletedAt and deletedBy
Response (200 OK)
{  "success": true,  "message": "Event soft deleted"}
4.7 Restore Soft Deleted Event
PATCH /api/v1/events/:id/restore
AccessAdmin only
Response (200 OK)
{  "success": true,  "message": "Event restored"}
4.8 Admin Approval (If Workflow Enabled)
PATCH /api/v1/events/:id/approve
AccessAdmin
Request Body
{  "approvalStatus": "approved | rejected"}
5. Dashboard-Specific Routes
5.1 Get Organizer Events
GET /api/v1/events/my
AccessOrganizer
Returns only:
Events where createdBy = logged-in user
Excludes soft-deleted
5.2 Get Archived Events
GET /api/v1/events/archive
AccessPublic or authenticated
Returns:
status = archived
isDeleted = false

4. Ownership Rules
Each event must contain:
createdBy  reference to User
Ownership Enforcement Rules:
Organizer can modify only events where createdBy equals their userId.
Organizer cannot modify events created by others.
Admin bypasses ownership restriction.
Student has no modification rights.
Ownership is validated in backend middleware before controller execution.
5. Event Lifecycle States
Each event must contain:
status (Enum):
draft
published
archived
Optional workflow layer:
approvalStatus (pending, approved, rejected)
6. Lifecycle Visibility Rules
Draft
Visible only to creator and admin
Not visible to students
Not included in public queries
Published
Visible to all users
Included in main event listings
Editable by owner or admin
Archived
Visible in archive section
Read-only
Cannot be modified by organizer
Admin modification allowed only if explicitly permitted
Soft Deleted
Not visible in any public query
Only visible to admin
Can be restored by admin
7. Lifecycle Transition Rules
Without Approval Workflow
draft  published
published  archived
Not Allowed
archived  draft
archived  published
With Approval Workflow Enabled
draft  pending approval
pending approval  approved  published
pending approval  rejected  draft
published  archived
8. Transition Authority
Draft  Published
If approval disabled:
Organizer (owner) or Admin
If approval enabled:
Admin approval required
Published  Archived
Organizer (owner) or Admin
Archived  Restore
Admin only (if policy allows)
Soft Delete
Organizer (owner) or Admin
Restore Soft Delete
Admin only
9. Soft Delete Behavior
Soft delete does not remove record permanently.
On soft delete:
isDeleted = true
deletedAt recorded
deletedBy recorded
System Behavior:
Excluded from all normal queries
Not visible in dashboards
Admin can view via special query
Admin can restore by setting isDeleted = false
Permanent deletion should be restricted to admin only, if implemented at all.
10. Archive Locking Policy
Archived events are locked.
Rules:
Organizer cannot modify archived event.
Organizer cannot revert archive state.
Admin may restore if policy allows.
Archived events cannot be deleted without admin authority.
Archive locking ensures institutional integrity.
